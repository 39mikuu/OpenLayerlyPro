## 门禁 1：Stripe Test Mode 全链路

所需凭证/前置：Stripe **Test Mode** secret key、webhook signing secret、可达的
公开（或隧道）webhook 回调地址。后台「Stripe」配置组（`STRIPE_GROUP`）填入测试
凭证，配置加密生效。

操作步骤（在隔离测试部署上，使用 Stripe 测试卡）：

1. 一次性 Checkout：完成支付 → 校验会员期开通、provider event inbox 记录、
   重复 webhook 幂等。
2. 自动订阅：开通 → 触发续费（invoice）→ 取消 → 校验周期推进、发票幂等、
   reconcile 对账一致。
3. 退款 / 拒付：发起全额退款与 dispute → 校验 reversal-first（先回收权益再处理
   交付）、重复 webhook 不重复扣权益。
4. 同用户并发 grant：人工 pending 与自动 pending 冲突 → 校验无会员时长丢失、
   无重复开通。

PASS：以上 4 组全部符合预期；日志/后台任务响应/delivery ledger 中**不出现**
Stripe secret 或原始 provider 错误体。

证据留存：Stripe Dashboard 测试事件 ID 列表、后台会员期截图、reconcile 结果、
脱敏日志片段。

## 门禁 2：真实 SMTP 可靠性

所需凭证/前置：真实 SMTP 服务（host/port/user/pass/from）。后台「SMTP」配置组
（`SMTP_GROUP`）填入并加密保存。

操作步骤：

1. 正常发送：触发登录码 / 通知邮件 → 收件箱确认到达，`Message-ID` 稳定唯一。
2. 临时失败（transient）：制造 4xx/超时 → 校验进入 defer 重试，后台可见重试。
3. 永久失败（permanent）：制造 5xx/无效地址 → 校验进入 dead，不无限重试。
4. 未配置 defer/dead：清空 SMTP → 校验业务不报错崩溃、任务安全 defer。
5. delivery ledger 与后台重发：人工重发一封 → 校验 ledger 记录与状态流转。

PASS：5 项行为符合分类；**日志、admin task response、delivery ledger、归一化
provider error 中均不出现验证码、SMTP secret、原始 provider 错误或原始收件人
地址**。

证据留存：脱敏后的 ledger 截图、Message-ID 样例、各失败分类的状态截图。

## 门禁 3：真实 CSP report-only → enforce 观察窗

所需凭证/前置：operator 真实外部来源全部接入——真实 Turnstile site key、真实
S3/R2 签名对象源、所有启用的公开 integration（脚本/连接/像素）、真实公开视频
嵌入域。相关后台配置组（`TURNSTILE_GROUP` / `STORAGE_GROUP` / integration）填好。

关键控制项：

- `SECURITY_CSP_MODE`：`auto`（默认；存在 legacy 可执行 footer 时自动保持
  report-only）/ `report-only`（强制观察）/ `enforce`（强制拦截）。
- 含 legacy 可执行 footer 时，`auto` 会保持 report-only 并在后台提示迁移；迁移
  或清除后才允许 `enforce`。

操作步骤：

1. 设 `SECURITY_CSP_MODE=report-only`，以真实流量/真实页面（公开、admin、登录）
   运行一个观察窗（建议 ≥ 数日或覆盖全部真实交互路径）。
2. 收集 CSP 违规报告，确认**没有**来自合法 Turnstile / 签名源 / 公开
   integration / 视频嵌入的误报阻断；如有则修正来源派生后重跑观察。
3. 若存在 legacy 可执行 footer：先按 `docs/handoff/harden-s6-security-response-headers.md`
   迁移/清除，后台迁移提示消除。
4. 切换 `SECURITY_CSP_MODE=enforce`，复验真实页面功能正常。
5. 用浏览器确认生产响应头：`script-src` **不含** `'unsafe-inline'` 与
   `'unsafe-eval'`；per-request nonce 唯一；HTTPS 部署设 `SECURITY_HSTS_ENABLED`
   后出现 `Strict-Transport-Security`。

PASS：观察窗无合法来源误报；enforce 后真实功能正常；生产 CSP 无 unsafe 指令。

证据留存：观察窗时间范围、CSP 报告样本、enforce 后页面功能截图、响应头抓取。

## 门禁 5：升级与恢复演练 + 外部密钥/恢复点托管

所需前置：受支持升级源 `v0.1.0` tag；本地存储与真实 S3/R2（或 MinIO 之外的
真实兼容端点）两套环境；异地保管的 `SESSION_SECRET`、`CONFIG_ENCRYPTION_KEY`。

操作步骤：

1. 升级演练：从 `v0.1.0` 按 `docs/deployment/upgrade.md` 执行——recovery point →
   停写 → pending-payment 修复 → one-off migrator → 强制 file-safety backfill →
   启动；校验迁移 12→18、配置与任务状态保留、`/api/ready` 绿。
2. 恢复演练（local 与真实 S3/R2 各一次）：
   ```bash
   ./scripts/backup.sh            # archive v2 + checksum
   # 人为制造 任务/支付/存储 漂移
   ./scripts/restore.sh ./backups/openlayerly-backup-<ts>.tar.gz
   ```
   校验：停 app/dispatcher 全程 → 校验和门禁 → schema 兼容探测（v1 fail-closed）→
   迁移 → 文件安全修复 → 任务/provider-event 中和 → DB↔存储收敛 → 启动后
   `/api/ready` 绿。S3/R2 对象字节需另由 provider snapshot/versioning 恢复。
3. 外部密钥语义：
   - 轮换 `SESSION_SECRET` → 旧会话失效（401），需重新登录。
   - 替换 `CONFIG_ENCRYPTION_KEY` → 加密配置不可解（如 `GET /api/admin/config/smtp`
     500），须用原 key 恢复。
   - 确认上述两个密钥、DB/归档、对象存储恢复点均**异地受控保管**。

PASS：升级与两套恢复演练 `/api/ready` 绿；密钥丢失/轮换语义符合预期；托管确认
到位。

证据留存：升级迁移日志、两份归档路径与 restore 输出、`/api/ready` 结果、密钥
托管确认（不含密钥本体）。
