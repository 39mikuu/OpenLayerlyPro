## External-secret semantics

- Rotating `SESSION_SECRET` invalidated an existing admin session (old cookie
  returned 401); users must sign in again and in-flight encrypted login-code
  tasks may become permanently failed.
- Replacing `CONFIG_ENCRYPTION_KEY` left the process syntactically ready but made
  an encrypted SMTP DB override undecryptable (`GET /api/admin/config/smtp`
  returned 500). The original key is required to recover encrypted settings.
- Neither secret is automatically recoverable from all deployment modes. The
  operator must preserve the exact external values separately.

## Unexecuted release blockers

The environment supplied no Stripe Test Mode, SMTP, real operator S3/R2,
Cloudflare Tunnel, or OpenAI-compatible provider credentials.

Therefore these mandatory checks remain unexecuted:

1. Stripe Test Mode one-time Checkout, subscription lifecycle, invoices,

2. Real SMTP success/transient/permanent/defer/dead/retry and delivery-ledger
   observation.
3. Operator's actual CSP report-only observation window followed by enforce,
   including actual Turnstile, signed storage origin and public integrations.
4. Real AI translation provider smoke test.
5. Operator confirmation of off-host `SESSION_SECRET`,
   `CONFIG_ENCRYPTION_KEY`, database/archive, and object-storage recovery-point
   custody.
6. GitHub Dependabot, Code Scanning/CodeQL and Secret Scanning alert review.
   Their REST endpoints returned 404 to the current token, so absence of alerts
   is not proven.

Until all six are evidenced and no blocker is found, the decision remains
**HOLD**. Do not create `v1.0.0`, publish a GitHub Release, or close #88/#64.

## Draft release notes

OpenLayerlyPro v1.0.0 combines manual payments, Stripe one-time and recurring
payments, auditable membership lifecycle, reliable mail/tasks, secure local/S3
file delivery, Markdown and multilingual publishing, AI-assisted translation,
nonce-based CSP/security headers, and hardened archive v2 recovery.

Upgrade from `v0.1.0` requires a complete recovery point, stopped writes,
pending-payment conflict remediation, the one-off forward migrator, and the
mandatory file-safety backfill before normal app startup. Downgrade migrations
are not supported; rollback restores the complete pre-upgrade DB/config/local
archive, previous image/source, matching object-storage recovery point, and
external secrets.
