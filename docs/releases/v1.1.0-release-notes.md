# OpenLayerlyPro v1.1.0 — Release Notes

Status: **released 2026-07-17** from merge commit `3a80b34`. The acceptance
evidence below was collected on 2026-07-16 from release-candidate builds and
retained as the release record.

## Highlights

OpenLayerlyPro v1.1.0 extends the v1.0 single-creator membership site into a
more complete writing, sponsorship, discovery, and operations release:

- multi-theme publishing: Blog and WordPress Classic themes, admin theme
  selection, static theme registration, shared Core view-models, G4 zh/en/ja
  i18n completeness, and G6 visual baselines across the landed themes;
- admin hardening: responsive shell/navigation, shared admin primitives,
  payment review fixes, safer editors and dialogs, mobile table patterns,
  accessibility/keyboard feedback, visual regression coverage, review
  follow-ups, and iOS Safari Chinese IME fixes;
- discovery and SEO: `/feed.xml` Atom output, sitemap index and shards,
  `robots.txt`, canonical/Open Graph/Twitter metadata for public content, and
  noindex/non-leaking metadata for restricted content;
- opt-in new-post notifications: default-off preferences, campaigns,
  per-recipient delivery ledger, task priority/reserved capacity, quotas,
  pacing, RFC 8058 unsubscribe headers, POST-first one-click unsubscribe,
  synchronous permanent-failure suppression, and G1 payload privacy;
- backup/restore safety for notifications: manifest format v4 records
  notification unsubscribe/suppression key IDs, sources, fingerprints, and
  file-backed key payloads; restore validates key continuity before replacing
  the production database and neutralizes restored notification work;
- supporter wall: default-off public `/supporters`, fan display names and
  opt-in controls, plain-text moderated dedications, effective-membership
  eligibility on every request, a bounded latest-200 approved-candidate public
  query, `/admin/supporter-wall`, and a required `SupporterWall` theme slot;
- analytics: optional Umami integration through the existing structured public
  integration and CSP revision path, with zero injection when unconfigured.

## Upgrading From v1.0.0

Before upgrading, create and verify a full recovery point using
`docs/deployment/upgrade.md` and `docs/deployment/backup-restore.md`. Keep the
old app image/source, database/config/local archive, matching S3/R2 recovery
point, and all external secrets until the v1.1.0 deployment and restore drill
are verified.

New migrations since `v1.0.0` are `0021` through `0027`:

- `0021_tired_human_fly.sql`: task claim and stale-lease indexes;
- `0022_public_feed_index.sql`: public feed keyset index;
- `0023_wp2_email_notifications.sql`: notification preferences, campaigns,
  deliveries, attempts, suppression, and related task state;
- `0024_g1_transactional_email_privacy.sql`: transactional email payload
  privacy migration;
- `0025_notification_delivery_fencing.sql`: delivery fencing and exact quota
  refund fields;
- `0026_wp5_supporter_wall.sql`: supporter wall entries;
- `0027_supporter_wall_admin_page_index.sql`: supporter wall admin paging index.

The normal entrypoint reruns forward migrations idempotently at startup. For a
controlled production upgrade, keep the app stopped and run the staged one-off
migrator first, then start the app only after the migration succeeds. The
in-place upgrade acceptance drill from `v1.0.0` is recorded under Acceptance
Evidence below.

## Environment and Secrets

v1.1.0 adds notification delivery and notification-secret configuration. The
important new environment keys are:

- queue and delivery pacing: `TASK_TRANSACTIONAL_RESERVED_PER_BATCH`,
  `TASK_NOTIFICATION_MIN_PER_BATCH`,
  `TASK_NOTIFICATION_STALE_RECLAIM_MAX_PER_BATCH`,
  `NOTIFICATION_EMAIL_DAILY_BUDGET`,
  `NOTIFICATION_EMAIL_PACING_PER_MINUTE`,
  `NOTIFICATION_CAMPAIGN_EXPANSION_BATCH_SIZE`,
  `NOTIFICATION_DELIVERY_MAX_AGE_HOURS`, and
  `NOTIFICATION_UNSUBSCRIBE_TOKEN_MAX_AGE_DAYS`;
- current notification keys: `NOTIFICATION_UNSUBSCRIBE_KEY_ID`,
  `NOTIFICATION_UNSUBSCRIBE_SECRET` or
  `NOTIFICATION_UNSUBSCRIBE_SECRET_FILE`,
  `NOTIFICATION_SUPPRESSION_DIGEST_KEY_ID`, and
  `NOTIFICATION_SUPPRESSION_DIGEST_SECRET` or
  `NOTIFICATION_SUPPRESSION_DIGEST_SECRET_FILE`;
- optional previous keys for rotation: `NOTIFICATION_UNSUBSCRIBE_PREVIOUS_*`
  and `NOTIFICATION_SUPPRESSION_DIGEST_PREVIOUS_*`.

Docker Compose production starts may leave the current notification secret
values unset: the entrypoint defaults key IDs to `current` and creates
persistent `0600` files at the `*_SECRET_FILE` paths. Previous keys are never
auto-generated; operators must configure and retain them explicitly until their
tokens or digests no longer need verification.

## Backup and Restore

New backups use `FORMAT_VERSION=4`. Archives that contain WP2 notification data
without v4 notification-key manifest fields fail closed before database
replacement, unless the compatibility probe proves there is no notification key
continuity data to protect.

For restores, supply the same file-backed notification key files or the exact
externally managed notification secrets whose fingerprints match the archive.
The restore preflight validates notification key source, key ID, SHA-256
fingerprint, and file mode before crossing the destructive boundary.

## Legacy Compatibility Deprecation Policy

This is the advance notice promised by the known-gaps ledger (G3). Three
legacy compatibility paths currently live in the codebase indefinitely:

- restore support for legacy `FORMAT_VERSION=1` backup archives (including
  the `--allow-legacy-v1-unknown-schema` escape hatch);
- the legacy raw-HTML footer migration path into the structured public
  integration/verification registry;
- the historical file-safety backfill (`files-backfill.mjs`) remediation
  for pre-v1.0 file data.

Original policy, announced with this release:

- All three paths remain fully supported for the entire **v1.1.x** line.
- Convert v1-format archives by restoring them on v1.1.x and creating a
  fresh backup (manifest v4). Run the documented upgrade flow (including
  the mandatory file-safety remediation) to clear the other two paths.
- Removal was initially targeted at v1.2.0, subject to at least 90 days of
  notice.

Policy amendment, 2026-07-17: v1.2 does not remove these paths regardless of
its release date. Removal is deferred to a separately scoped release no earlier
than 2026-10-14 (currently expected to be v1.3). Instances that still depend on
these paths must upgrade or restore through a v1.1.x release before moving to
the eventual removal release.

## Out of Scope

v1.1.0 does not add a generic Plugin runtime, Hub discovery product, theme
marketplace, third-party theme lifecycle, comments, PPV/tips, video
transcoding/thumbnails/HLS, self-hosted analytics dashboards, or default
telemetry.
