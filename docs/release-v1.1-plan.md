# v1.1 计划书：不只画师

> 状态：**提案，待创作者确认**。本文档定义 v1.1 的目标、范围、工作包、顺序与发布门槛。
> 前置条件：`v1.0.0` 正式发布（#104 验收完成）+ 发布后 4–6 周稳定窗口（该窗口内只接受 bug 修复与文档）。
> 决策记录约定沿用现有机制：涉及架构边界的决策落 ADR，功能取舍以本文档与对应 issue 为准。

## 1. 版本主题与背景

**一句话主题：不只画师——写作者与赞助创作者也能用它拥有自己的会员站。**

决策依据：

1. **创作者本人成为第一个真实用户。** 维护者将以「博客 + 爱发电式赞助」场景运营自己的正式实例。v1.1 的每个工作包都直接服务这一真实使用，这是需求不跑偏的第一道保险。
2. **扩大适用人群，而不改变定位。** v1.1 不触碰「单创作者、自托管」的定位与 Core 边界；它把同一套系统从「画师作品站」延伸到「写作者/博客/赞助站」，目标用户外延变大，产品内核不变。
3. **全部构建在已有基座上。** 六个工作包分别复用 Theme v1 契约、S5 邮件可靠性与 durable outbox、S6 CSP 动态来源、配置中心与 i18n——没有一项需要新的架构层。

本版本**明确不做**的内容见 §4。

## 2. 范围：六个工作包

优先级即编号顺序。WP6 为溢出项：进度紧张时最先砍。

### WP1 博客主题 + 后台主题选择器

- **用户故事**：作为把站点当博客用的创作者，我需要一个文字排版优先的主题；作为任何创作者，我需要在后台切换站点主题。
- **范围内**
  - 第二个内置主题「Blog」：文章列表流（标题 + 摘要 + 日期，弱化封面网格）、正文阅读排版（行宽、行高、目录可后置）、暗色模式与现有字体/颜色预设兼容。
  - 后台「外观」页新增活动主题选择器：站点级单选，立即生效，记录 audit 事件。
  - 两个主题共用同一套 Core view-model 契约；Blog 主题不得引入新的数据需求，如确有需要，先扩展契约并同步内置主题。
- **范围外**：主题包上传/安装、主题市场、per-post 主题、第三方主题生命周期（维持 roadmap ⏸ 状态）。
- **依赖基座**：Theme v1 数据契约与 view-model 边界（`docs/architecture/theme-system.md`）。
- **验收**
  - [ ] 全部公开页面在 Blog 主题下渲染正常，权限矩阵（匿名/登录/会员/admin）无回退。
  - [ ] 明暗模式、颜色预设、自定义 hue、zh/en/ja 在两个主题下均正常。
  - [ ] 主题切换不需要重启，切换行为有 audit 记录。
  - [ ] Theme 边界不变：主题不接触权限、数据库或服务端 secret。
- **预估**：大（本版本最大项，约占整体一半）。

### WP2 新内容邮件通知

- **用户故事**：作为粉丝/读者，我希望关注的创作者发布新内容时收到邮件；作为创作者，我希望发布行为自动触达订阅者，带来回访与续费。
- **范围内**
  - 用户侧通知偏好：粉丝账号新增「新内容邮件通知」开关（默认关闭，显式 opt-in）。
  - 发布触发：内容首次发布（含定时发布到点）时，为每个 opt-in 且有权查看该内容可见级别的用户入队一封通知邮件；member 内容只通知当期有效会员。
  - 邮件经由现有 durable outbox / task 管线投递，遵守 S5 失败分类、重试与 delivery ledger；按收件人 `users.locale` 本地化（zh/en/ja）。
  - 幂等：同一 post 同一用户至多一封；重新发布/编辑不重复触发（首次发布语义）。
  - 本地化：邮件框架文案按收件人 `users.locale`；文章标题/摘要优先使用该 locale 的
    已发布译文，无则回落原文。
  - 退订：邮件内一键退订链接（签名 token，不要求登录；token 与用户绑定、可长期有效
    但在用户重新开启通知后失效旧 token）。
  - **投递额度与信誉治理（本 WP 的硬性不变量）**：
    - 双通道优先级：事务邮件（登录码、付款、续费提醒）与通知邮件在 outbox 中分道；
      预算、限速、defer 只作用于通知道，**通知在任何情况下不得饿死或延迟事务邮件**。
    - 每日通知预算：配置中心新增每日发送上限（默认保守值，按 UTC 日重置），超出部分
      复用 S5 defer 语义顺延次日，不丢弃、不计为失败。单实例语义；多实例共享预算
      属 HA 线非目标。
    - 限速 pacing：通知按可配置速率（每分钟 N 封）匀速投递，即使预算充足也不突发，
      保护自建 SMTP 的 IP/域名信誉。
    - campaign 惰性展开：发布时仅落一条 campaign 任务，收件人由 dispatcher 按预算
      分页展开；不产生无界事务，支持中途取消。
    - 硬退信抑制：永久失败地址自动进入抑制列表，后续群发跳过，控制退信率。
    - 文档：SPF/DKIM/DMARC 配置指引与自建邮件服务器批量投递的信誉预热提示；
      明确引导「订阅更新首选 RSS（WP3），邮件通知为补充」。
- **范围外**：站内通知中心、Web Push、分标签订阅；每日/每周**摘要模式**（数量级最大的
  额度压缩器，但需聚合状态——观察项：出现日更且订阅量大的真实部署者再立项）。
- **依赖基座**：S5 邮件可靠性、durable task/outbox、定时发布、`users.locale`。
- **验收**
  - [ ] 可见性矩阵：public/login/member × 匿名/登录/会员 的通知对象集合正确。
  - [ ] 定时发布到点触发一次且仅一次；取消定时不触发。
  - [ ] SMTP 故障时进入 S5 defer/dead 语义，不阻塞发布事务。
  - [ ] 退订链接无需登录即生效，且不泄露收件人身份给第三方。
  - [ ] 群发规模有界（campaign 惰性分页展开），单次发布不产生无界事务。
  - [ ] 通知道预算耗尽/限速/defer 期间，登录码与付款邮件仍即时投递（双通道回归测试）。
  - [ ] 预算超限的通知顺延次日自动继续，不丢失、不重复。
  - [ ] 硬退信地址进入抑制列表后不再收到群发，但事务邮件不受抑制列表影响。
- **预估**：中。

### WP3 RSS / Atom 输出

- **用户故事**：作为读者，我用 RSS 阅读器订阅这个博客；作为创作者，我获得零维护的分发渠道。
- **范围内**
  - `/feed.xml`（Atom）：仅 public 内容，标题 + 摘要（或安全渲染后的正文，二选一，倾向摘要）、发布时间、规范链接；按站点默认语言输出，译文以 alternate link 呈现。
  - 有界条目数（如最近 20 条）、正确的 `Cache-Control`、条目 GUID 稳定。
- **范围外**：member 内容的带 token 私有 feed（等真实需求）、podcast 扩展、JSON Feed。
- **验收**
  - [ ] 主流阅读器（Miniflux/Feedly）解析通过；W3C validator 无错误。
  - [ ] login/member 内容绝不出现在 feed 中（含摘要与图片 URL）。
  - [ ] 输出经过与页面一致的 HTML 安全处理，不引入新的 XSS 面。
- **预估**：小。

### WP4 内容 SEO：sitemap + 分享卡片 + hreflang

- **用户故事**：作为创作者，我的公开文章应该能被搜索引擎正确索引，被分享到社交平台时有像样的卡片。
- **范围内**
  - `sitemap.xml`：public 内容 + 公开列表页，含 `lastmod`；有界分页。
  - 文章级 meta：`og:title/description/image`（封面图缺省回落站点 logo/og 图）、`twitter:card`、canonical。
  - 已发布译文输出 `hreflang` alternate（PRD §7 遗留项收口）；`robots.txt` 显式化。
  - login/member 内容：`noindex`，不进 sitemap。
- **范围外**：结构化数据 JSON-LD 全家桶（只做 Article 基础字段或留到后续）、站长平台自动提交。
- **验收**
  - [ ] Google/Bing 校验工具通过；非 public 内容在 sitemap 与 meta 层面均不可发现。
  - [ ] 三语译文的 hreflang 互指正确，与 UI 语言协商不冲突。
  - [ ] 分享到 X/Discord/Telegram 的卡片渲染正确（真实抓取验证）。
- **预估**：小到中。

### WP5 赞助者鸣谢墙

- **用户故事**：作为赞助者，我希望被看见；作为创作者，我希望公开展示支持者作为社交证明。
- **范围内**
  - **粉丝显示名设置（前置依赖）**：`users.displayName` 目前仅在管理员初始化时写入，
    粉丝没有任何设置入口。本 WP 需在粉丝账号页新增显示名编辑（长度/字符校验、
    XSS 安全处理）。展示规则：未设置显示名的会员即使 opt-in 也不展示，
    **任何路径都不得回落到邮箱或其他身份信息**。
  - 公开「支持者」页：展示 opt-in 且已设置显示名的当期有效会员（显示名 + 等级，
    不显示金额与邮箱）。
  - 会员侧开关：默认**不展示**，用户显式同意；随会员过期自动下墙，反转（退款/拒付）立即下墙。
  - 创作者侧：可隐藏整页、可移除单个展示（不解释义务）。
- **范围外**：金额排序/打赏榜（与「会员制」语义冲突，且诱导攀比）、历史赞助者永久墙、
  **赞助者留言**（等价于一个迷你评论系统，带反垃圾与审核负担，与评论非目标同一理由拒绝）。
- **依赖基座**：会员生命周期（active/suspended/revoked）、Theme 契约（新增一个公开页 view-model）。
- **验收**
  - [ ] 默认不泄露任何会员身份；opt-in 状态变化即时生效并有 audit 记录。
  - [ ] 未设置显示名的会员不出现在墙上；任何渲染路径不回落邮箱。
  - [ ] 显示名输入经过与站点其他用户输入一致的安全处理（长度、字符、转义）。
  - [ ] 会员过期/反转后不再展示（含缓存路径）。
  - [ ] 两个主题下渲染正常，zh/en/ja 文案完整。
- **预估**：小到中（含粉丝显示名设置）。

### WP6（溢出项）自托管访客统计集成：Umami / Plausible

- **用户故事**：作为创作者，我想看站点访问量，但不想把访客数据交给第三方巨头。
- **范围内**：配置中心新增「Analytics」组（脚本 URL + site id，DB 加密存储）；S6 CSP 动态来源放行该域；仅公开页面注入；后台「集成」页状态呈现；一页部署文档（Umami compose 示例）。
- **范围外**：自建统计、仪表盘内嵌、任何默认开启的数据上报。
- **验收**
  - [ ] 未配置时零注入、零请求；配置后强制 CSP 下正常工作。
  - [ ] 管理页不回显完整密钥；audit 记录配置变更。
- **预估**：小。**进度紧张时首先砍掉，顺延 v1.2。**

## 3. 顺序与里程碑

```text
M0  v1.0.0 发布 + 稳定窗口（4–6 周，只修不加）
M1  WP3 RSS + WP4 SEO           ← 小件先行，创作者实例当天受益
M2  WP1 博客主题 + 主题选择器    ← 本版本主体
M3  WP2 邮件通知                 ← 依赖真实 SMTP 验证，放在主题之后
M4  WP5 鸣谢墙 (+ WP6 如有余量)
M5  v1.1 验收与发布
```

M1–M5 的目标总体量为 **6–8 周**（不含 M0 稳定窗口）。超出预期时的收缩顺序：
先砍 WP6，再将 WP5 顺延 v1.2；WP1–WP4 是本版本的不可减内核。

原则：每个 M 合并进 main 后创作者实例即升级试用（持续 dogfooding），发现的问题在下一个 M 前优先处理。

## 4. v1.1 明确不做（非目标重申与新增）

| 项 | 状态 | 理由 / 触发条件 |
|---|---|---|
| OAuth / 第三方登录 | 挂起 | 非目标清单现役条目；目标用户实际可用的 provider（微信/QQ）个人自托管者无法申请。触发条件：多个真实部署者指名其粉丝实际使用的 provider。登录摩擦先以 magic link（后续候选）解决 |
| 评论系统 | 不做 | 反垃圾/审核/法务是单人维护沼泽；替代：mailto / 外链社交串。若未来社区强需求，应为 Plugin 生态候选而非 Core |
| 一次性打赏（无等级赞助） | 观察 | 收款码人工审核流程可覆盖；创作者实例运营中若被证实别扭再立项 |
| 视频封面/缩略图/时长 | 顺延 v1.2 候选 | 画师线第一体验缺口，但与本版本「写作者」主题不同线；避免版本失焦 |
| 主题包上传 / 主题市场 | 维持 ⏸ | 属 Plugin 级生命周期问题 |
| Plugin runtime / Hub / HA | 维持 roadmap 触发条件 | 见 roadmap.md |

## 5. 发布门槛（v1.0 纪律的轻量版）

- [ ] 全部 WP 验收项通过；`pnpm test / lint / format:check / tsc / build` 与既有 CI 门禁绿。
- [ ] 权限矩阵回归：新增的 feed、sitemap、鸣谢墙、通知路径均纳入跨切面授权测试。
- [ ] 新增公开端点（feed/sitemap/支持者页）完成有界性检查（条目数、分页、请求体）。
- [ ] 邮件通知在真实 SMTP 环境完成投递/失败/重试/退订验证。
- [ ] zh/en/ja 三语文案完整（UI、邮件、新公开页）。
- [ ] 升级验证：从 v1.0.0 实例原地升级 + 备份恢复演练通过（新增表/配置纳入 archive 契约）。
- [ ] CHANGELOG、PRD、roadmap、admin/deployment 文档同步更新。
- [ ] 发布文案（三语）：官网新增 v1.1 亮点或博客文章。

## 6. 风险与对策

| 风险 | 对策 |
|---|---|
| WP1 主题工作量失控（第二主题挖出契约缺口） | 先做「契约盘点」半日预研；缺口先补契约再写主题；列表流以现有 keyset 分页为准，不新造查询 |
| WP2 烧穿第三方 SMTP 额度，挤掉登录码等事务邮件 | 双通道优先级 + 每日通知预算 + defer 顺延；事务邮件永不受预算约束 |
| WP2 群发损耗自建 SMTP 的 IP/域名信誉（进垃圾箱） | 限速 pacing + 硬退信抑制列表 + 严格 opt-in/一键退订；文档提示 SPF/DKIM/DMARC 与预热；RSS 作为首选订阅通道分流 |
| 稳定窗口涌入 v1.0 修复挤占排期 | 窗口内不启动任何 WP；v1.1 起点以窗口结束为准，而非日历日期 |
| 单人维护并行过多 | 严格按 M1→M5 串行；WP6 预设为可砍 |

## 7. 成功信号（发布后 4 周复盘）

- 创作者自己的实例以博客主题日常运营，且愿意继续用（最硬的信号）。
- 出现 ≥1 个非画师身份的真实部署者（写作者/播客/赞助站）。
- 邮件通知打开数 > 0 且退订率不异常（自己实例的 delivery ledger 即可观察，无需遥测）。
- showcase / Discussions 中出现主题相关反馈（无论好坏，有反馈即有人在用）。

复盘产出：v1.2 主题的立项依据（当前候选：视频媒体体验线，或本清单第三梯队中被真实使用顶上来的项）。
