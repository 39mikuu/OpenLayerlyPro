## 待立项

### G1 `tasks.payload_json.to` 明文存储收件人邮箱

- **现状**：邮件任务 payload 中直接保存收件人地址；`release-v1.0-checklist.md` §3
  已承认该软点，将保护责任交给数据库与备份运维。
- **风险**：数据库/备份泄露即泄露全量收件人；v1.1 WP2 邮件通知会使该表敏感数据量显著放大。
- **方向**：payload 中收件人改为 user id 引用（发送时解引），或字段级加密；
  历史行提供一次性脱敏 backfill。
- **建议时机**：**随 v1.1 WP2 一并实施**（同一批代码、同一次验收）。
- **体量**：中。

### G2 `monthlyCharLimit` 保存但不强制

- **现状**：AI 翻译的月字符预算仅保存与展示，不做本地强制（PRD §Phase 7 自注）。
- **风险**：诚实性缺口——创作者会误以为存在成本保护。
- **方向**：二选一：实现本地用量记账与强制；或在后台 UI 明确标注「仅记录，不限制」。
- **建议时机**：半天体量，任意稳定窗口收割。
- **体量**：小。

### G3 legacy 兼容面没有退场时间表

- **现状**：v1 archive 探测、legacy footer 迁移、历史文件 backfill 等兼容代码
  无限期存活，构成永久维护税。
- **方向**：制定并公告弃用政策（例：v1 archive 支持至 v1.2；更旧实例须先升级到
  v1.x 再跳）。弃用宣布得越早成本越低。
- **建议时机**：**v1.0 发布公告中一并宣布**，代码移除按公告的时间表执行。
- **体量**：政策宣布小；后续移除各自小。

### G4 三语 i18n 缺少完整性门禁

- **现状**：zh/en/ja 的 key 对齐依赖人工；UI、API 错误、邮件模板三个面都可能漂移。
- **风险**：v1.1 将新增大批文案（通知邮件、鸣谢墙、博客主题），漂移概率随之放大。
- **方向**：CI 脚本校验三语 key 集合一致（缺失/多余均报错），纳入现有检查序列。
- **建议时机**：**v1.1 的首个功能 PR 合并前铺好**；允许在同一 WP1 PR 中落地，但不得把门禁本身推迟到 WP1 完成后。
- **体量**：小。

### G5 CI actions 的 Node 20 弃用警告

- **现状**：`checkout@v4`、`configure-pages@v5` 等被 runner 强制运行于 Node 24
  并打印弃用警告（2026-07 Pages 部署日志可见）。
- **方向**：下次修改 workflow 时顺手升级各 action 到当前主版本；仍须保持第三方 action 使用不可变 commit SHA。
- **建议时机**：搭车任意 workflow 改动，不单独立项。
- **体量**：微。

### G6 主题缺少视觉回归基线

- **现状**：公开页面无截图对比测试；当前单主题下 CSS 改动靠人眼验收。
- **风险**：v1.1 WP1 引入第二主题后，任何样式改动都需要双主题人工核验。
- **方向**：Playwright 截图基线（关键公开页 × 明暗模式 × 两主题），阈值比对。
- **建议时机**：**WP1 第二主题合并前**铺好基线；可以与第二主题放在同一 PR，但不能把第二主题标记完成后再补。PR #123 在补齐该门禁前只能保持 Draft。
- **体量**：小到中。
